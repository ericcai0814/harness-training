Slide 06 — Function Calling 4-Step
06 / 16 · ANATOMY

這頁要花時間講，因為這是 99% 的人對 FC 的最大誤解。聽到 function calling 三個字大家以為模型在「呼叫」函數，像我們寫程式那樣函數真的被執行，不是的。

模型只回傳一個結構化的字串說「我想呼叫 get_weather('Taipei')」，是你的程式去執行這個呼叫、拿到結果、塞回對話，模型本身連網路都沒連、連檔案系統都沒碰，它只會「想」跟「說」。

這個誤解一旦修正，harness 在 Step 3 可以做的事就清楚了——驗證參數、拒絕執行、加 approval、記 transcript、rate limit，整個 hook、approval、quality-gate 都是長在 Step 3。

底部那句「工具回傳的內容本身就是一個提示詞」講的是同一件事，你 Step 4 塞回去的不是純資料，是新一輪的 context，會直接影響模型下一輪怎麼想。

Slide 07 — One sentence to remember.
07 / 16 · DIVISION

這句金句要記下來，後面 Agent Loop 跟 Hook 都會回扣。

模型只負責「決定要做什麼」這一件事——它「想」（reasoning），它「說」（輸出 tool_calls），你的 harness 負責三件事：真的去做（執行）、決定能不能做（approval / hook）、結果怎麼餵回去（context management）。

下次有人問你 Agent 是什麼、LLM 系統是什麼，這句話就是一句話的總結，記住這個分工，後面 Part 4、Part 5 就是這個分工在不同層的展開。

Slide 08 — Agent Loop = while
08 / 16 · LOOP

這就是 chatbot 跟 agent 的根本差別，chatbot 一問一答結束，agent 是個 while 迴圈會自己跑很多輪直到完成任務。

Claude Code 跟你互動時背後就是這個迴圈在跑，每一圈：模型看 messages、決定要呼叫工具還是直接回答、如果呼叫工具就 harness 執行加 append、進下一圈，圈數可能 3 可能 30 看任務複雜度。

這個迴圈要理解，因為後面 Part 5 講的 4 個新問題（context 爆炸、失憶、跑偏、驗證）全部都是 loop 跑久了的副作用，你寫 hook、你管 context、你做 approval，都是在這個迴圈的不同位置插門。

Slide 09 — Why context engineering matters
09 / 16 · STATELESS

這頁是 context engineering 的根本原因，模型本身沒有「上一輪我做了什麼」這個概念，它每一圈都是把整個 messages 陣列從頭讀一遍。

它不記得自己上一圈想了什麼，只能從前一圈的輸出反推，所以 context 管理才那麼重要，你該記得的東西必須塞進每一輪的 messages，不然模型就忘了。

Claude Code 的 auto-compact、prompt cache、各工具明確的 result format，全部都是在處理「模型每圈從零讀」這件事，下次你的 agent「忘記」剛才討論過的東西，先檢查 context 是不是被截掉了，不是它笨。

Slide 10 — 4 problems = 4 engineering tracks
10 / 16 · SIDE EFFECTS

這 4 個問題不是 bug，是 loop 本身帶出來的副作用，理解 Part 4 的 while loop 之後，這 4 個就是必然會出現的：

每圈塞 context 必然爆、模型沒記憶必然失憶、自己跑必然會偏、自己說完必然要被驗證，Harness Engineering 大半工作就在解這 4 個。

對 ewill 來說你已經踩到至少 3 個：寫 quality-gate 是解第 4 個（驗證）、寫 hook 是解第 3 個（約束）、想用 sub-agent 是解第 1 個（context 切分）。

問題是這 4 個你們目前各自為戰、一個一個 case 處理，今天看完希望你把它們當同一件事的不同切面來想，這才有可能變成團隊規範。

Slide 11 — Capabilities + Hooks
11 / 16 · WHERE TO INSERT

ewill 看到 Hook 應該最有共鳴，我們的 quality-gate 就是 Post-Hook（編輯完跑 lint、type check、test），stop-orchestrator 也是某種 Hook，重點是你在用了但還沒系統化看待它。

Hook 不是「自動化」，是在迴圈裡插一個門讓某些事不會跑掉，Pre-Hook 是「能不能做」的門，Post-Hook 是「做完算不算數」的門。

三能力裡記憶機制 ewill 也在做，你寫 CLAUDE.md 就是給 agent 記憶。「沒有記憶的 agent 像新來的實習生」這個比喻試著想像，每天早上來上班的同事都不記得昨天做過什麼，整個團隊一週就崩了。

所以 CLAUDE.md 不是「設定檔」是「給實習生交接的筆記」，下次寫的時候從這個角度切，會比較知道要寫什麼、不寫什麼。

Slide 12 — You vs The Stack
12 / 16 · LAYERS

這頁是要讓 ewill 看見 CC 不神奇，它每一輪丟給模型的 messages 大概長右邊那樣，是好幾層東西組起來的：

Slide 13 — Live Trace · 4 rounds
13 / 16 · TRACE

這頁重點在看 model 跟 harness 各自負責什麼。

Round 1 model 想要 grep、model 說 tool_call，是 harness 真的去 grep；Round 2 同樣模式讀檔；Round 3 最關鍵，model 想 edit 加跑測試，Pre-Hook 會在 edit 真的下去之前先檢查一道（例如目錄白名單、危險指令攔截），Post-Hook 會在跑完之後再做一道（例如 lint、type check、test 全跑）。

這兩個 hook 不是模型決定要不要跑、是 harness 寫死在迴圈裡的，下次看到別人寫的 hook 記得問一句：這個 hook 插在 loop 的哪個位置？Pre 還是 Post？防止什麼事跑掉？

Slide 14 — Without harness = ChatGPT 2022
14 / 16 · COUNTERFACTUAL

這頁是反過來證明 harness 的價值，如果把上面 8 個東西拿掉，剩下的就是裸模型加一個 chat 介面，那就是 ChatGPT 2022 年剛出來時候的水準。

從 2022 到 2026 模型本身有變強，但主要的 UX 跳躍其實是這 8 個 harness 元素堆起來的，回扣最開始那句「模型負責想和說，你的程式負責做和餵」 — 過去 4 年的進步主要不是「想」變強，是「做和餵」的工程做得越來越完善。

對 ewill 來說，你的 harness（CLAUDE.md、hook、quality-gate、skill）每加一塊整體系統就強一塊，而且這些東西模型版本升級不會作廢，模型換、你的 harness 還在，這比追工具划算很多。

15 / 16 · DECISION

這頁回答柏崴上次提的「機制邊界不清」 — 「skills、CLAUDE.md、Hook 之間的使用邊界不夠直觀」。今天我們把判準講清楚。

判準三步驟。第一：物理上要擋住、不能被繞過嗎？要擋就用 Hook，PreToolUse 擋下、PostToolUse 加工。第二：是條件性多步流程嗎？想避免每次對話都吃 token 嗎？是就用 Skill。第三：全域行為偏好、要每次都套用嗎？是就用 Rule。

特別講 Skill 的「揭露式注入」 — 這是它跟 Rule 最大差別。Rule 寫進 CLAUDE.md，每次跟 Claude 對話它都讀一次，token 永遠付。Skill 不一樣，只有描述常駐（最多 1,536 字元），主體要用到才載入。所以 Rule 適合一定永遠都要的事、Skill 適合「條件性才需要」的事，譬如 PR review 流程，平常根本用不到，寫進 CLAUDE.md 就是浪費 token。

Skill 還有兩個進階能力 — 一是可以包 Hook，但這個 Hook 只有 skill 啟動時生效，不像獨立 Hook 一直開著。二是可以注入動態 context，譬如 !`gh pr diff` 會先跑 shell 命令、把結果替換進 prompt，Claude 看到的是已展開的版本。這兩個 Rule 完全做不到。

看下面三個案例。「不要讀 .env」 — 要物理擋、零容忍，這是 Hook。「commit 用 zh-TW Conventional Commits」 — 全域行為偏好、沒有特定 tool 邊界，這是 Rule。「PR Review 流程」 — 多步、需要動態注入 diff、平常不用，這是 Skill。

特別記中間那句 — Rule 適用就不要寫 Hook。維護成本 Rule < Skill < Hook，能用便宜的層就用便宜的。下一段行動共識，每人挑一件回去試的事，順便指認落在三層哪一層。
