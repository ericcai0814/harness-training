01 · Introducing hooks
STATION 01 · CONCEPT

Hook 是什麼?

Hook = 在 Claude 動手用 tool 之前 或 之後,插一段你寫的程式碼。

PreToolUse
Tool 跑之前
可以擋下(exit 2)或放行。模型看不到 tool 結果,只看到你的 stderr 訊息。
PostToolUse
Tool 跑之後
擋不了,但可以加工(format / lint)或把結果丟回去給模型參考。
配置三層 · LIKE CLAUDE.md
  • ~/.claude/settings.json — 全域(所有 project)
  • .claude/settings.json — 團隊共享(會 commit)
  • .claude/settings.local.json — 個人(不 commit)
6 種應用場景
  • 自動 format
  • 自動跑測試
  • 擋讀敏感檔
  • 跑 linter
  • log Claude 動作
  • 檢查命名規範
常見問題
Hook 跟 CLAUDE.md 規則差在哪?
CLAUDE.md 告訴模型該怎麼做 → 模型可能無視。Hook 程式碼物理擋下來 → 模型繞不過。
三層 settings 同時存在誰贏?
local > shared > global(由近到遠覆蓋,跟 CLAUDE.md 一樣)。
跟 permissions.deny 差在哪?
deny 是 declarative 名單;Hook 是 imperative 程式碼,可寫任意邏輯。
寫 hook 用什麼語言?
任何 shell 能跑的東西,範例用 Node.js,Python/bash 都行。
Hook = 物理攔截,不是禮貌請求 01 / 09
Hook 講白了就是在 Claude 動手用工具之前或之後,插一段你寫的程式碼。之前的叫 PreToolUse — tool 還沒跑,你可以擋下來,回一個錯誤訊息給模型看。之後的叫 PostToolUse — tool 已經跑了你擋不了,但可以做後續動作,比如自動把剛改的檔 format 一下、自動跑測試、把結果丟回去給模型參考。配置上跟 CLAUDE.md 一樣有三層:全域、團隊共享、個人本機。第 5 站 init 示範會看到 settings.local.json 是怎麼產生的。Hook 能做的事很多,這頁列了六種,第 7 站 Useful hooks 會挑兩個真實案例細講。旁邊 Q&A 有些常見疑問先列在這,等等講完語法你應該都會自動懂。

寫 hook 就 4 步

照著做就完成了。每一步對應 settings.json 裡一個欄位 — 沒有黑魔法。

STEP 1 選 Pre 還是 Post tool 之前還是之後
STEP 2 選監控哪個 tool Read / Write / Bash · matcher 支援 regex
STEP 3 寫 command stdin 收到 JSON · 任何 shell 能跑
STEP 4 exit code + stderr 回應給 Claude · stderr 變模型訊息
EXIT CODE 規則
  • Exit 0 — 放行,tool 正常跑
  • Exit 2 — 擋下(只 PreToolUse 有效),stderr 內容變成 Claude 看到的訊息

    "file_path": "/code/queries/.env"  ← 判斷重點

常見問題
stdin JSON 的 schema 我怎麼知道?
第 8 站會教 jq trick — 先寫一個 log hook 把 stdin 印出來看,再寫真的 hook。
command 一定要 Node.js 嗎?
不用。type: "command" 直接交給 shell,可以跑 python script.py、bash 任何能執行的東西。
PostToolUse 為什麼擋不了?
tool 已經跑完了,沒得擋。Post 的能力是「加工」或「丟訊息回模型」。
matcher 可以寫複雜 pattern?
支援 regex,例如 "Write|Edit" 一次盯多個 tool。
同一 tool 多個 hook 誰先誰後?
settings.json 裡定義的順序就是執行順序。
Pre / matcher / command / exit · 四件事而已 02 / 09
上一站講 hook 是什麼,這站看 hook 怎麼寫。寫 hook 就 4 步,照著做就完成了。第一步,選 Pre 還是 Post — 你想在 tool 跑之前介入還是之後。第二步,選你要盯哪個 tool。Read、Write、Bash 這些。matcher 也可以寫 regex,比如 Write|Edit 一次盯兩個。第三步,寫一個 command。Claude Code 會把 tool call 的 JSON 從 stdin 餵給你 — 中間這個範例就是 stdin 長相。重點看 tool_input 那段,你想擋什麼就在那邊判斷,比如這個範例就是模型想 Read /code/queries/.env — 你的 command 解 JSON、看到 .env 就決定擋下來。第四步,用 exit code 回應。Exit 0 = 放行,Exit 2 = 擋下。Exit 2 的時候你 stderr 寫的訊息會被 Claude 看到。要注意 Exit 2 只有 PreToolUse 有效。可以盯哪些 tool 看右下角這個列表。但 — MCP server 加進來會多新 tool,最簡單的辦法是直接在 Claude Code 裡問「列出可用工具」。下一站 Implementing a hook 我們把這 4 步串起來,寫一個真的能跑的 hook,就是擋讀 .env 那個範例。
Station 03Implementing a hook First example
03 · Implementing a hook
STATION 03 · CODE

任務 — 擋住 Claude 讀 .env

Hook 不是一支 script,是兩個檔案 — 一個註冊表 + 一支邏輯 script。

FILE 1 · .CLAUDE/SETTINGS.LOCAL.JSON
JSON{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Read",
        "hooks": [
          {
            "type": "command",
            "command": "node $PWD/hooks/read_hook.js"
          }
        ]
      }
    ]
  }
}

↑ $PWD 下一站講

FILE 2 · HOOKS/READ_HOOK.JS
JAVASCRIPTprocess.stdin.setEncoding("utf8");
let input = "";
process.stdin.on("data", (d) => (input += d));
process.stdin.on("end", () => {
  const toolArgs = JSON.parse(input);
  const readPath =
    toolArgs.tool_input?.file_path || "";
  if (readPath.includes(".env")) {
    console.error("You cannot read .env");
    process.exit(2);   // ← 擋下
  }
  process.exit(0);   // ← 放行
});
SCHEMA 對照 · 為什麼只擋 READ,不擋 BASH / GREP?
Read.tool_input → { "file_path": "..." }
Bash.tool_input → { "command": "..." } ← schema 不同
Grep.tool_input → { "pattern": "...", "path": "..." }

→ 要全面擋:每個 tool 寫一個 matcher,或用 permissions.deny: ["Read(**/.env)"] 跨 tool 統一

常見問題
$PWD 怎麼變絕對路徑?
下一站 Gotchas 講,那是 init script 的功勞。
要連 Bash cat .env 也擋?
兩條路 — ① 多寫 matcher: "Bash" parse command 字串 ② permissions.deny "Read(**/.env)" 一次擋全部。
optional chaining ?. 必要嗎?
不是,但建議。tool_input 不同 tool 結構不同,加上比較不會爆。
為什麼用 includes 不用嚴格 path?
簡單 demo。production 改 path.basename 或 minimatch 比較精準。
被 hook 擋下,模型反應?
看到 stderr 訊息,知道為什麼被擋,會 replan 跳過或改用其他工具。
5 行 JS · 1 個 matcher · hook 完成 03 / 09
前面看了概念跟語法,這站開始寫真的程式碼。任務:擋住 Claude 讀 .env。注意,hook 不是一支 script,是兩個檔案 — 一個註冊表加一支邏輯 script。第一個檔,.claude/settings.local.json,是 hook 的註冊表。它告訴 Claude Code「監控 Read tool,每次模型要 Read 的時候,跑 read_hook.js 這支」。matcher 寫 Read,type 是 command,command 那一行 node $PWD/hooks/read_hook.js,那個 $PWD 等等下一站 Gotchas 講為什麼這樣寫。第二個檔,hooks/read_hook.js,邏輯就五行 — 從 stdin 讀 JSON,parse,看 tool_input.file_path 有沒有 .env,有就 stderr 寫訊息加 exit 2 擋下來,沒有就 exit 0 放行。這支 hook 只擋 Read 一個 tool。如果 Claude 改用 Bash 跑 cat .env 就過了。為什麼?因為每個 tool 的 tool_input schema 不同 — 看下面這三行:Read 是 file_path、Bash 是 command、Grep 是 pattern + path。要全面擋有兩條路 — 寫多個 matcher 各自處理,或用 permissions.deny 一次擋住跨 tool。Production 環境通常兩個都用:hook 寫客製邏輯,permissions.deny 守底線。這支 hook 第 6 站會直接跑給你看,現在先把 code 結構記在腦袋裡。

常見問題
相對路徑為什麼不安全?
path interception — 攻擊者放同名 script 在 PATH 較前位置;binary planting — 把惡意 script 放進 cwd 讓你誤執行。
直接寫死絕對路徑不用 $PWD?
可以,但 settings.json 就只能你自己用,team 沒法 commit 共享。
settings.local.json 為何 .gitignore?
含的是你機器的絕對路徑,commit 上去隊友 clone 下來會壞。example 才 commit。
$PWD 是 shell 變數,init script 怎處理?
字串 replace — 把 "$PWD" 字串直接換成 process.cwd() 的結果,不是真的展開 shell 變數。
example commit · local gitignore · setup 串起來 04 / 09
上一站看了 hook 的 code,但跑起來之前有個小坑要先處理。等等第 5 站 init 示範你會看到 .claude 目錄底下有兩個 settings.json,這站先解釋為什麼。起點是 Claude Code 官方的安全建議:hook command 要用絕對路徑,不要用相對路徑。為什麼?兩個攻擊向量 — path interception 跟 binary planting。簡單講就是攻擊者可以放假腳本騙你執行,相對路徑就被解析到攻擊者的 script。絕對路徑能避開這風險。但絕對路徑帶來新問題 — 我機器的 hook 路徑跟你機器的不一樣,settings.json 沒法直接 commit 給 team 用。每個人 clone 下來路徑都不同。解法是 placeholder + setup script。寫一份 settings.example.json,command 那邊用 $PWD 當 placeholder,這個版本可以 commit 進 repo。然後寫一支 setup script,每個人 clone 後跑 npm run setup,裡面的 scripts/init-claude.js 會把 $PWD 字串換成本機 process.cwd() 的結果,產生 settings.local.json。這個帶你機器絕對路徑的版本 .gitignore 不 commit。注意 — 那個 $PWD 不是真的 shell 變數展開,是 init script 用字串 replace 換掉的。下一站 init 示範我會跑 npm run setup 給你看,settings.local.json 從 example 變出來的過程。

常見問題
"You cannot read .env" 模型懂嗎?
懂。模型把 stderr 當 system feedback,通常會 replan 跳過或改用別的工具。
改成 "用 cat 看 .env" 呢?
模型會用 Bash,這支 hook 沒擋 Bash → 過了。這正是 Slide 3 講的 schema 不同問題。
Hook 失敗(crash)怎麼辦?
Claude Code 會顯示 hook 異常,多數情況仍放行 tool(避免你被自己的 bug 鎖在外面)。
stderr 會被算進 context 嗎?
會。所以訊息要短而精準 — 不然一個 hook 攔截就吃掉幾百 token。

Agent SDK 啟動的 sub-Claude 是獨立 session,不繼承原 session 的 hook config。

可以。Agent SDK 是 official 接口,自己 call API 或用其他模型做 review,pattern 一樣。

解法 · META HOOK 把 STDIN 印出來

  {
    "matcher": "*",
    "hooks": [{
      "type": "command",

    }]
  }
]
// ↑ jq . = pretty-print stdin

先 log · 後寫 · 工程習慣 08 / 09
前 7 站都在講 PreToolUse 跟 PostToolUse,這站把「其他 7 種 hook」跟「debug 不熟 hook 的方法」一起講完。先看左上 — Claude Code 一共有 9 種 hook event。Pre/Post 是最常用的兩種,剩下 7 種各自有用途,這站不展開細講每種,只要把握「Pre/Post 不是全部」這個概念就好。Q&A 那邊我有列每種的典型用途,回去查就有。但這引出寫 hook 的真實麻煩 — stdin 結構會變。看右邊兩個 stdin 範例:第一個 PostToolUse + TodoWrite,多了 tool_response 跟 oldTodos / newTodos;第二個 Stop hook,只有 stop_hook_active 一個 boolean,連 tool_input、tool_response 都沒有。每個 hook event + matcher 組合都長不一樣,文件不一定寫齊全,版本變動還會新增欄位。寫 hook 時你不能假設 stdin 長什麼樣,要實際看。解法:寫一個 logger hook 把 stdin 印出來看。看下半的 settings 片段 — matcher 寫 *(任何 tool 都觸發),command 一行 jq . > pre-log.json。jq 是 CLI 的 JSON parser,jq . 就是 pretty-print stdin。一行就配好了,不用寫 .js script。注意 — 這 demo repo 的 settings.example.json 已經配好了兩個 logger,Pre 跟 Post 都有。所以我們前面跑了 init 示範、read_hook 示範、query_hook 示範這些之後,repo 根目錄就會有 pre-log.json 跟 post-log.json。我現場 cat 一下 — cat pre-log.json。看,這就是真實 PreToolUse 的 stdin,剛剛 read_hook 觸發時 stdin 收到的就是這個結構,跟 Slide 2 我給的範例完全對得起來。所以寫新 hook 的 workflow 是 — 一,配 logger 跑幾次目標 hook event。二,cat log 看真實 stdin 結構。三,根據實際欄位寫真正的邏輯。四,寫完了再把 logger 撤掉,或留在 dev 環境用。比看文件可靠很多。先 log 後寫是 production 工程師寫 hook 的標準手法。這就把 hook 教學 8 站走完了。從 Slide 11 那一句「Pre 之前 / Post 之後」展開到 9 種 hook event、4 個 production case、加 1 個 meta hook,加 init 跟 read_hook 跟 query_hook 三個 live demo。後面接 Slide 14 反證跟 Slide 15 落地優先級,收尾整場。
